# Day 28 — Progressive Syncs

Progressive Syncs: A feature that controls the order and pace at which Applications generated by an ApplicationSet are synced. Instead of syncing all generated Applications simultaneously, you can define a rollout strategy that syncs them in stages.

RollingSync Strategy: The primary progressive sync strategy. It defines `steps`, where each step specifies which Applications to sync (using `matchExpressions` label selectors) and how many can sync at once (`maxUpdate`). Applications in a later step only begin syncing after all Applications in the previous step are Healthy.

AllAtOnce Strategy: The default behavior. All generated Applications sync simultaneously with no ordering or gating.

Enabling Progressive Syncs: Requires the `applicationsetcontroller.enable.progressive.syncs` feature flag to be set to `true` in the `argocd-cmd-params-cm` ConfigMap.

# --- 1. ENABLE PROGRESSIVE SYNCS ---
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cmd-params-cm
  namespace: argocd
data:
  applicationsetcontroller.enable.progressive.syncs: "true"
```
```bash
# Restart the ApplicationSet controller to pick up the config
kubectl -n argocd rollout restart deployment argocd-applicationset-controller
kubectl -n argocd rollout status deployment argocd-applicationset-controller
```

# --- 2. PROGRESSIVE SYNC — ROLLINGSYNC STRATEGY ---
# Syncs dev first, then staging, then production
```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: progressive-guestbook
  namespace: argocd
spec:
  goTemplate: true
  goTemplateOptions: ["missingkey=error"]
  generators:
  - list:
      elements:
      - env: dev
        namespace: guestbook-dev
      - env: staging
        namespace: guestbook-staging
      - env: prod
        namespace: guestbook-prod
  strategy:
    type: RollingSync
    rollingSync:
      steps:
      # Step 1: Sync dev first
      - matchExpressions:
        - key: env
          operator: In
          values:
          - dev
        maxUpdate: 100%
      # Step 2: After dev is Healthy, sync staging
      - matchExpressions:
        - key: env
          operator: In
          values:
          - staging
        maxUpdate: 100%
      # Step 3: After staging is Healthy, sync production
      - matchExpressions:
        - key: env
          operator: In
          values:
          - prod
        maxUpdate: 1
  template:
    metadata:
      name: 'guestbook-{{.env}}'
      labels:
        env: '{{.env}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/argoproj/argocd-example-apps.git
        targetRevision: HEAD
        path: guestbook
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{.namespace}}'
      syncPolicy:
        automated:
          selfHeal: true
        syncOptions:
        - CreateNamespace=true
```

# --- 3. VERIFICATION ---
```bash
kubectl apply -f progressive-appset.yaml

# Watch the rollout progress
argocd appset list
argocd app list

# Dev syncs first, then staging, then prod
argocd app get guestbook-dev
argocd app get guestbook-staging
argocd app get guestbook-prod
```

Operational Insight
Progressive Syncs bring deployment safety to fleet-scale GitOps. Without them, a bad config change pushed to Git would instantly deploy to every cluster simultaneously. With RollingSync, you get a built-in canary pattern: deploy to dev first, verify it's healthy, then roll to staging, and finally to production. The `maxUpdate` field lets you control blast radius—for production, you might set `maxUpdate: 1` so clusters are updated one at a time, giving you time to catch issues before they spread.
